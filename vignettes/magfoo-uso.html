Librería magfoo

Usar la función load para “cargar” el contenido de archivos, y la de save para almacenar datos en archivos, puede producir la sorpresa desagradable de haber perdido datos de manera inadvertida debido a sus efectos colaterales.

Antes de platicar el problema que se trata de resolver con las funciones en magfoo, conviene, como antecedente, recordar cómo es que las aplicaciones usan la memoria RAM.

Todo lo que es la ejecución de los programas y el procesamiento ocurre en el CPU. Pero este está compuesto de sólo unos cuantos compartimientos llamados registros, donde se hacen las operaciones. Todas las miles o millones de “variables” que utilizan las aplicaciones durante su ejecución están alojadas en la memoria RAM, no en el CPU. Para que el CPU tenga acceso a datos específicos que están en la RAM, en un área dedicada de ella, denominada stack, hay lo que se llama la tabla de símbolos, que es, esencialmente, una especie de libreta de direcciones donde están anotadas los nombres de las “variables” (símbolos) y la dirección dentro de la RAM donde están los datos que le corresponden. Cada vez que el CPU requiere algún dato en particular, hace un llamado a la tabla de símbolos para consultar en cuál lugar de la RAM está el dato que necesita, y con eso resuelto, lleva el dato en cuestión a sus registros para hacer la operación (suma, multiplicación, o lo que sea) que el código le pide que haga.

Cuando en R se manda ejecutar la línea de código que viene adelante, ocurren dos cosas:

  1. en algún lugar de la RAM se almacena el valor 2.3
  2. en la tabla de símbolos se anotan el símbolo (o nombre) x, y la dirección (posición) del dato dentro de la RAM. En el argot de la programación a esto se le llama, en inglés, “binding”; es decir, la asociación del símbolo con el dato por intermedio de la dirección. Desde otra perspectiva, también se dice que x “apunta” al valor 2.3, o que 2.3 fue asignado a x.
x <- 2.3

Es por medio de los nombres que el programador se refiere a los datos que están en la RAM, y hace que R evalúe las instrucciones para obtener un resultado; por ejemplo

log(x)
## [1] 0.8329091

La librería lobstr permite explorar algunas de estas cosas y aprender lo que ocurre “tras bambalinas”. La siguiente instrucción devuelve la dirección (dentro de la RAM) donde está el valor asociado al símbolo x. Es costumbre que las direcciones sean reveladas en números de base 16 (número hexadecimal) en lugar de la familiar base 10.

library(lobstr)
obj_addr(x)
## [1] "0x237fd64e020"

Vea lo que pasa ahora

x <- 2.3
obj_addr(x)
## [1] "0x237fcd44e10"
x <- 2.3
obj_addr(x)
## [1] "0x237fcd67898"

Eso muestra que los procesos de asignación del valor (en este caso, 2.3) en la RAM y del nombre o símbolo, son independientes. Lo que pasa es que cada vez que se le pide a R hacer un “binding”, este le “pide” al sistema operativo que administra la RAM un lugar para guardar el dato, y este asigna el espacio conforme a cierto algoritmo que da como resultado direcciones diferentes.

En el siguiente caso, se abre una entrada en la tabla de símbolos para alojar y, y se copia la dirección donde apunta x. Como resultado, R dice que x e y son objetos idénticos.

x <- 2.3
y <- x
obj_addr(x)
## [1] "0x237fdc3ff88"
obj_addr(y)
## [1] "0x237fdc3ff88"
identical(x, y)
## [1] TRUE

Ahora,

y <- y + 1
obj_addr(x)
## [1] "0x237fdc3ff88"
obj_addr(y)
## [1] "0x237fcc659e0"
identical(x, y)
## [1] FALSE

Al modificar el valor asociado a y, el sistema operativo ubica al resultado en otra dirección y los objetos x e y dejan de ser idénticos.

“Subir” o “cargar” el contenido de archivos

Por lo dicho antes, debe ser obvio que un mismo símbolo o nombre sólo está vinculado con una misma posición en la RAM, de manera que si en la libreta de direcciones el nombre mi_df refiere un conjunto de datos, y se mandara a leer lo que contiene un archivo con la instrucción load, y en ese archivo hubiera cualquier objeto que, al momento de haber sido almacenado con la instrucción save, tuviera el nombre mi_df, lo contenido en el archivo con ese nombre va a “sustituir” al conjunto de datos al que “apuntaba” mi_df antes de mandar a ejecutar load. Por ejemplo:

## archivo temporal para el ejemplo
tf <- tempfile(fileext = "rda")
mi_df <- 0

print(mi_df)
## [1] 0
save(mi_df, file = tf)

## si después, en la misma u otra sesión de trabajo
## (que puede ser varios días después, cuando ya ni
##  recuerda lo que almacenó en el archivo)
## define un nuevo objeto
mi_df <- data.frame(x = 1:4, y = 4:1)
print(mi_df)
##   x y
## 1 1 4
## 2 2 3
## 3 3 2
## 4 4 1
## e inadvertidamente "carga" el contenido del archivo
load(tf)

## lo que tendrá es lo que estaba almacenado en el archivo
## y "perderá" el data.frame creado líneas atrás
print(mi_df)
## [1] 0
## eliminar el archivo temporal del ejemplo
unlink(tf)

La solución que se ofrece en el “help” de load es “subir” los datos a un nuevo “environment”, evitando así el conflicto de nombres; pero eso vuelve un poco más engorroso referirse a los datos. Otras soluciones que utilizan la funciones readRDS y saveRDS, sólo permiten manejar un objeto en el archivo, y, como se afirma en el “help” de readRDS, el intercambio por internet es problemático.

Manejo de lo que se almacena en archivo

Un problema con save es que el almacenamiento no es acumulativo, en el sentido de que cada vez que se utiliza el mismo nombre de archivo este es recreado, de manera que lo que ya estaba almacenado en él, se pierde. Eso obliga a usar archivos con diferentes nombres (en especial con saveRDS) y en consecuencia, a la proliferación de archivos.

Funciones en magfoo

Las funciones en magfoo facilitan el almacenamiento y la “subida” de datos a la RAM, evitando los inconvenientes que acarrean load y save. Con ellas es posible administrar cuantos data.frame, vectores o listas genéricas se quiera, utilizando un solo archivo; y también, manejar de modo más seguro la “subida” de datos del archivo a el “ambiente” de trabajo global (el que usa R por defecto). Además hay una función que mejora la funcionalidad de la función comment, para agregar metadatos a lo que se almancene en los archivos.

La implementación de las funciones se hizo de manera sencilla, por medio del uso conveniente de la opción envir de load y save.

  1. list_off y list_dff: revelan los nombres de los objetos almacenados en un archivo, junto a sus metadatos asociados. La primera produce un listado de todos los objetos; la segunda, sólo el de los data.frame. Las funciones no “suben” los datos.
  2. read_off y read_dff: “suben” los objetos o data.frame con el mismo nombre con el que están almacenados.
  3. get_off, get_dff, get_dffs: “subir” un data.frame u objeto, y asignarle un nuevo nombre en el ambiente de trabajo. Hace un “binding”. En el caso de get_dffs, se produce un objeto lista, cuyos elementos son los data.frame mandados a “subir”.
  4. add_tof, save_add y save_df: para almacenar de manera incremental objetos o tablas de datos a un archivo. Con save_df se manda al archivo un solo data.frame, pero con la posibilidad de indicar un nuevo nombre con el que será guardado. Eso es útil porque es usual que uno asigne nombres crípticos, muy cortos, durante el trabajo interactivo, pero desea que sea almacenado con un nombre más largo pero más descriptivo.
  5. rm_off: para eliminar, de manera selectiva, objetos del archivo.
  6. meta: sustituta de comment para asignar y leer metadatos, el que puede ser, inclusive, el nombre de un archivo tipo texto donde se registran metadatos más extensos que una simple línea.

Unos ejemplos:

library(magfoo)

ADA <- tempfile("ej", fileext = ".rda") #archivo temporal para ejemplo

w <- data.frame(par = c(0, 2, 4), impar = c(1, 3, 5))

meta(w) <- "primeros 3 num. im(p)ares" # asigna un metadato
save_df(w, "im_par", ADA) # almacena con nombre im_par
## 
## ... d.f im_par agregado a C:\Users\eddy\AppData\Local\Temp\RtmpKsy2cB\ej5c30194120fc.rda
primo <- c(1, 2, 3, 5)
meta(primo) <- "primeros num. primos"
add_tof(primo, file = ADA) #agrega el vector con el mismo nombre
## [1] "primo"
una_lista <- list(impar = w, primo = primo)
add_tof(una_lista, file = ADA)
## [1] "una_lista"
list_off(ADA)
##        data                      meta      class
## 1    im_par primeros 3 num. im(p)ares data.frame
## 2     primo      primeros num. primos    numeric
## 3 una_lista                      <NA>       list
list_dff(ADA)
##     data                      meta
## 1 im_par primeros 3 num. im(p)ares
read_dff(im_par, file = ADA) #carga de data.frame con el mismo nombre
## [1] "im_par"
jaja <- get_dff(im_par, file = ADA) #carga asignando otro nombre
meta(jaja) #metadato asignado
## [1] "primeros 3 num. im(p)ares"
rm_off(primo, file = ADA) #elimina el vector
## Warning in rm_off(primo, file = ADA): 
## !!!! ... primo removido de C:\Users\eddy\AppData\Local\Temp\RtmpKsy2cB\ej5c30194120fc.rda

##        data                      meta      class
## 1    im_par primeros 3 num. im(p)ares data.frame
## 2 una_lista                      <NA>       list
unlink(ADA) #elimina archivo temporal
